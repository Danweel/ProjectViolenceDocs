Contributing to Project Violence Documentation

This guide covers how to set up your environment and how to maintain the project.

# Quick Start for Contributors

## Prerequisites

> Python 3.12+
> Poetry (for Python dependency management)
> Git
> VSCodium (recommended) or VS Code

## Steps:

### 1. Clone the Repository

```
git clone https://github.com/Danweel/ProjectViolenceDocs.git
cd ProjectViolenceDocs
```

### 2. Set Up the Environment

Run the setup script provided in the root directory. It handles dependencies and environment creation automatically.

```
chmod +x setup_for_contributors.sh
./setup_for_contributors.sh
```

> Note: If you are on Windows, ensure you are using Git Bash or WSL to run this script.

### 3. Verify the Setup

Run the following to ensure everything is working:

```
poetry env info --path
poetry run python --version
poetry run sphinx-build -b html docs/source docs/_build/html
```

If the build succeeds, open the documentation locally:


   `xdg-open docs/_build/html/index.html`


## Maintenance & Updates

This section covers how to safely update project components.

### Directory Structure Reference
```
ProjectViolenceDocs/
├── `pyproject.toml`          # Poetry config (dependencies live here)
├── `poetry.lock`             # Locked dependency versions, autogenerated
├── `requirements.txt`        # Autogenerated
├── projectviolencedocs-*/    # Virtual environment (auto-generated, DO NOT EDIT)
├── build/                    # Generated HTML (deletable/regeneratable)
└── docs/
    ├── `Makefile`            # Sphinx build rules
    ├── `make.bat`            # Automatically generated
    └── source/
        ├── _templates/       # html templates (404)
        ├── `index.rst`       # Documentation root
        ├── `conf.py`         # Sphinx configuration
        ├── user/             # User-facing docs (.rst files)
        └── _static/
            ├── images/       # screenshots
            ├── site pngs     # favicon, logo-light, logo-dark
            └── `custom.css`


```

### Updating Mermaid Diagrams

The Mermaid JavaScript library is pinned to a specific version in docs/source/conf.py for stability reasons.

#### When to update:
- A new Mermaid feature is released.
- Security vulnerabilities are reported.
- The CDN becomes unreliable for the pinned version.

#### How to update:
1. Check the latest stable version: https://github.com/mermaid-js/mermaid/releases
2. Update mermaid_version in docs/source/conf.py:

```
    mermaid_version = '11.13.0'  # Replace with the new version in the conf.py
```

3. Test locally first:

    rm -rf docs/_build
    poetry run sphinx-build -b html docs/source docs/_build/html

4. Verify all diagrams render correctly.

5. Then commit the change:

    git add docs/source/conf.py
    git commit -m "Update Mermaid to vX.Y.Z"

### Updating Sphinx and Extensions

Dependencies are managed via Poetry in pyproject.toml.

**How to update**:

1. Check for outdated packages:

    poetry show --outdated

2. Update a specific package:

    poetry update sphinx

3. Update all packages (use with caution):

    poetry update

4. Test the build and review changes:

    git diff pyproject.toml poetry.lock

5. Regenerating requirements.txt

If you add or update dependencies, regenerate requirements.txt for Read the Docs:

    poetry export -f requirements.txt --output requirements.txt --without-hashes --only docs

## Troubleshooting Basic Issues

Below are some basic errors I encountered during development and their solutions (as a person that didn't fully know what they were doing at first).

### **`Could not resolve interpreter path` in VSCodium**

VSCodium shows `Default interpreter path ... could not be resolved`.

**Cause**: The Poetry environment folder name changed (hash updated), but .vscode/settings.json still points to the old path.

**Solution**:

1. Run `poetry env info --path` to get the new path.
2. Update .vscode/settings.json with the new path.
3. Reload VSCodium (Ctrl+Shift+P → Developer: Reload Window).

---

### **`ModuleNotFoundError: No module named 'encodings'`**

Poetry commands fail with a fatal Python error (i.e., one that prevents the build)

**Cause**: This usually happens if running inside a Flatpak sandbox that cannot access system Python libraries. (or if you've really confused things, some other environement, like a .venv you started manually, which causes a similar issue with Poetry, see below)

**Solution**:

- Preferred: Install the native .deb version of VSCodium instead of Flatpak.
- Workaround: Run commands in your native terminal, not inside the Flatpak terminal.

---

### **`poetry env list` shows nothing but prompt shows venv**

Cause: Shell prompt is stale; venv was created outside Poetry's control.

Fix:

1. Deactivate

   `deactivate`

2. Find where the venv actually is

   `poetry env info

3. Delete it manually

   `rm -rf /path/from/env/info`

4. Reinstall

   `poetry install --with dev,docs`

---

### **`Mermaid error: Requires mermaid js version X or later`**

Build fails with a Mermaid version mismatch.

**Cause**: The mermaid_version in conf.py is older than what the Python package expects, or the CDN version is missing.

**Solution**:

1. Update `mermaid_version` in docs/source/conf.py to the latest stable version (e.g., 11.13.0).
2. Clear the build cache: `rm -rf docs/_build`.
3. Rebuild: `poetry run sphinx-build -b html docs/source docs/_build/html`.

---

### **"Extension error: Could not import extension..."**

Build fails with `No module named 'sphinx-...'`.

**Cause**: The extension is listed in conf.py but not installed in the Poetry environment.

**Solution**:

1. Install the package: `poetry add <package-name>`.
2. Ensure the import name in conf.py matches the package (e.g., `notfound.extension` for `sphinx-notfound-page`).

---

### **"Explicit markup ends without a blank line" (RST Errors)**

Warnings in usage.rst or other .rst files.

**Cause**: Missing blank lines after blocks (e.g., `.. note::`) or malformed links.

**Solution**:

- Ensure there is a blank line after every directive.
- Check indentation (RST is strict about spaces vs. tabs).
- Verify links follow the correct format for text, local files, other Sphinx docs and external links all have slightly different formats.

---

### **"Theme error: no theme named 'furo' found"**

Build fails because the theme is missing.

**Cause**: Package listed in poetry.lock but not actually installed in the venv.

**Solution**:

Install the theme:

```
    poetry add groundwork-sphinx-theme.
```
Ensure html_theme = 'groundwork' is set in conf.py.

This can be tricky, because sometimes the theme works even when this error pops up. I think this may be because of linting or other error checking not understanding some of the formatting in the Sphinx-based conf.py.

Check if the theme is actually installed, Furo as the example:

    `poetry run python -c "import furo; print(furo.__version__)"`

Besides suppressing the error, there's not much you can do about it.

---

### **"Valid: False" in `poetry env info`**

**Cause**: Virtual environment is corrupted (interrupted install, crash, file permission issues).

**Fix (Nuclear Rebuild)**:

   `cd ~/Documents/VSCodiumFiles/ProjectViolenceDocs`

1. Deactivate any active shell

   `deactivate 2>/dev/null || true`

2. Manually delete the corrupted environment folder

   `rm -rf projectviolencedocs-*-py3.12`

3. Delete the lock file

   `rm poetry.lock`

4. Reinstall everything

   `poetry install --with dev,docs`

5. Verify
```
   poetry env info  # Should show Valid: True
   poetry run python -c "import furo; print('OK')"
```

This is a bit silly but there's a script included `fix-env.sh` that does the above, I messed with it so many times. Learning curve issues.

---

**`Command not found: sphinx-autobuild`**

**Cause**: Poetry is creating a new, empty virtual environment in the current directory instead of using the project's venv.

**Fix**:

1. Ensure you're in the PROJECT ROOT (where pyproject.toml is)

   `cd ~/Documents/VSCodiumFiles/ProjectViolenceDocs`

2. Run from there with full paths

   `poetry run sphinx-autobuild docs/source docs/build/html --port 8001`

**Prevention**: Always be in the root folder before running `poetry run` commands. Sounds obvious but it's an easy mistake. That's probably all this error ever means.
